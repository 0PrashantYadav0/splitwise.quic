Splitwise-QUIC — Architecture

A Splitwise clone served entirely over HTTP/3 + QUIC, with real-time updates pushed as QUIC DATAGRAM frames over WebTransport. Below are the system, request lifecycle, QUIC upgrade, real-time, and data diagrams.

Go 1.26 HTTP/3 · QUIC WebTransport HTMX + Tailwind SQLite (pure-Go) quic-go 0.60

1 · System architecture

The browser bootstraps over HTTP/2 (TCP), then upgrades to HTTP/3 (QUIC) via the Alt-Svc header. A separate WebTransport session carries live updates as QUIC datagrams.

2 · Request lifecycle (add an expense)

A POST mutates the DB, logs activity, and broadcasts an event that refreshes every connected client's balances in real time. 

sequenceDiagram
    autonumber
    actor U as User (browser)
    participant H3 as HTTP/3 server
    participant MW as Auth middleware
    participant HD as createExpense handler
    participant SP as splits.Compute
    participant ST as store
    participant DB as SQLite
    participant HUB as realtime.Hub
    participant O as Other clients

    U->>H3: POST /g/{id}/expenses (over QUIC)
    H3->>MW: route + verify session cookie
    MW->>HD: authorized request
    HD->>HD: parse amount, currency, split inputs
    HD->>SP: Compute(total, splitType, inputs)
    SP-->>HD: per-user shares (exact sum)
    HD->>ST: CreateExpense(expense, shares)
    ST->>DB: INSERT expense + shares (tx)
    DB-->>ST: ok
    HD->>ST: LogActivity(...)
    HD->>HUB: Publish(expense event)
    HUB-->>O: SSE "update" + QUIC datagram
    O->>H3: hx-get /balances (auto refresh)
    HD-->>U: 303 redirect -> group page

3 · TCP → QUIC upgrade & handshake

How a browser discovers and switches to HTTP/3, including 0-RTT resumption on reconnect. 

sequenceDiagram
    autonumber
    participant B as Browser
    participant T as TCP :4433 (HTTP/2)
    participant Q as UDP :4433 (QUIC/HTTP3)

    Note over B,T: First visit
    B->>T: TLS 1.3 handshake + GET /login
    T-->>B: 200 OK + Alt-Svc: h3=":4433"; ma=2592000
    Note over B: Caches the h3 advert

    Note over B,Q: Subsequent requests
    B->>Q: QUIC Initial (1-RTT) + TLS 1.3 (ALPN "h3")
    Q-->>B: handshake + session ticket (for 0-RTT)
    B->>Q: HTTP/3 requests (multiplexed streams)

    Note over B,Q: Reconnect later
    B->>Q: QUIC 0-RTT (early data, no extra RTT)
    Q-->>B: accept early data
    Note over B,Q: Connection migration:
QUIC keeps the session across IP changes

Key takeaways

  • One port, two transports: TCP (h1/h2) and UDP (h3/QUIC) both on :4433.
  • Two live channels: QUIC datagrams (WebTransport) as the fast lane, SSE as the fallback.
  • Pure money math: the splits package has zero I/O and is fully unit-tested.
  • Derived balances: never stored — always computed from expenses + settlements per currency.
  • Self-contained: templates & assets embedded; pure-Go SQLite; single binary.
